## Business Rules

scaffold-be はこのセクションを正として実装する。AC はこのルールの具体例（テストケース）。

### 状態遷移

エンティティにライフサイクル（ステータス）がある場合、遷移図で定義する。

```
{状態A} ──{操作X}──→ {状態B}
{状態B} ──{操作Y}──→ {状態C}
{状態B} ──{操作Z}──→ {状態A}（差し戻し）
```

#### 遷移条件マトリクス

| 現在の状態 | 操作    | 遷移先  | 前提条件 | 失敗時                     |
| ---------- | ------- | ------- | -------- | -------------------------- |
| {状態A}    | {操作X} | {状態B} | {条件}   | {エラー/無視}              |
| {状態B}    | {操作Y} | {状態C} | {条件}   | {エラー/無視}              |
| {状態B}    | {操作Z} | {状態A} | {条件}   | {エラー/無視}              |
| {状態A}    | {操作Y} | —       | —        | {エラーコード}（不正遷移） |

#### 初期状態

- 新規作成時: `{初期状態}`

#### 終了状態

- `{終了状態}`: 以降の全操作を禁止。変更不可。

### 権限モデル

ロール（自分/パートナー/管理者等）ごとに、エンティティに対する操作可否を定義する。
scaffold-be は各 service 関数の冒頭で権限チェックを実装する。

#### ロール定義

| ロール     | 判定方法                    | 説明               |
| ---------- | --------------------------- | ------------------ |
| 所有者     | {entity}.createdBy = userId | レコードの作成者   |
| パートナー | {partner解決ロジック}       | 所有者のパートナー |
| 第三者     | 上記以外                    | 権限なし           |

#### 操作権限マトリクス

| 操作            | 所有者 | パートナー | 第三者 | 備考                   |
| --------------- | ------ | ---------- | ------ | ---------------------- |
| 作成            | ○      | ○          | ×      |                        |
| 閲覧            | ○      | ○          | ×      |                        |
| 更新            | ○      | ×          | ×      | ステータス={状態A}のみ |
| 削除            | ○      | ×          | ×      | ステータス={状態A}のみ |
| {操作X}         | ○      | ×          | ×      |                        |
| {操作Y}（承認） | ×      | ○          | ×      | パートナーのみ実行可能 |

### エンティティ間制約

エンティティ間の参照関係と所有権ルール。scaffold-be は service 層で検証する。

| 親エンティティ | 子エンティティ | 制約                                                     | 違反時               |
| -------------- | -------------- | -------------------------------------------------------- | -------------------- |
| {Category}     | {Item}         | 作成・更新時に userId が一致すること                     | 403 FORBIDDEN        |
| {User}         | {Expense}      | 紐づく支払者が世帯メンバーであること                     | 400 VALIDATION_ERROR |
| {Parent}       | {Child}        | 親が削除可能状態であること（子が{状態}の場合は削除不可） | 409 CONFLICT         |

### 計算ロジック

金額集計や按分等の計算ルールがある場合、数式で定義する。

- **合計額**: `SUM(items.amount)` WHERE {条件}
- **按分**: `{金額} / 2`（2人等分）
- **表示フォーマット**: `¥${amount.toLocaleString()}`

### 外部API連携

外部サービスと連携する場合、成功・失敗パターンを全て定義する。

| パターン                         | HTTP/ネットワーク    | DB更新 | 通知 | レスポンス      |
| -------------------------------- | -------------------- | ------ | ---- | --------------- |
| 成功                             | 200 OK               | ○      | ○    | 201             |
| API エラー（4xx）                | 400/401/403          | ×      | ×    | 502 BAD_GATEWAY |
| API エラー（5xx）                | 500/502/503          | ×      | ×    | 502 BAD_GATEWAY |
| ネットワークエラー               | タイムアウト/DNS/TLS | ×      | ×    | 502 BAD_GATEWAY |
| 設定なし（APIキー未設定等）      | —                    | ○      | ×    | 201（成功扱い） |
| 送信先なし（ユーザーID未設定等） | —                    | ○      | ×    | 201（成功扱い） |

#### トランザクション境界

- 外部API呼び出し **より前** のDB更新はトランザクション内で行う
- 外部API失敗時はロールバックする
- 設定なし・送信先なしの場合は DB更新をコミットし、通知をスキップする

### 境界条件・エッジケース

scaffold-be が見落としやすい境界条件を列挙する。

| 条件                               | 期待する動作                              | 対応AC |
| ---------------------------------- | ----------------------------------------- | ------ |
| ロール未設定（null）のユーザー     | パートナー解決結果 = null、関連操作は不可 | AC-XXX |
| 存在しないIDの指定                 | 404 NOT_FOUND                             | AC-XXX |
| 同一ユーザーが自分のレコードを承認 | 403 FORBIDDEN（自分の承認は不可）         | AC-XXX |
| 月パラメータが不正（2026-13等）    | 400 VALIDATION_ERROR                      | AC-XXX |
| 並行操作の競合（同時チェック等）   | 後勝ち（楽観的）                          | AC-XXX |
