# テスト実行と自動修正

テストを実行し、失敗した場合は原因を分析して修正する。

## 前提条件

- 実装コードが存在すること
- テストコードが存在すること

## 引数

| 引数   | コマンド                                                                  | 用途                                     |
| ------ | ------------------------------------------------------------------------- | ---------------------------------------- |
| `unit` | `npm run test:unit -- --run && npm run test:integration -- --run`         | scaffold 直後（E2E テストがまだない段階） |
| `all`  | `npm run test`                                                            | scaffold-test-e2e 完了後                 |

引数が省略された場合は `unit` として扱う。

## ワークフロー

### 1. テストコマンドの判断

引数に応じてテスト実行コマンドを決定する。引数が不明な場合は `specs/infra-spec.md` またはプロジェクトの設定ファイル（package.json, Makefile, pyproject.toml 等）を参照して判断する。

### 2. テスト実行

引数 `unit`（デフォルト）の場合:
```bash
npm run test:unit -- --run && npm run test:integration -- --run
```

引数 `all` の場合:
```bash
npm run test
```

### 3. 失敗時の分析

- エラーメッセージを確認
- スタックトレースから問題箇所を特定
- 期待値と実際の値の差分を分析

### 4. 修正の実装

**テストコードは仕様であるため、原則として編集しない。** 失敗原因に応じて対処:

| 失敗原因                       | 対処                                         |
| ------------------------------ | -------------------------------------------- |
| 実装のバグ                     | 実装を修正                                   |
| モックの設定ミス               | モック設定を修正                             |
| テストの期待値が仕様と食い違う | **実装を仕様に合わせる**（テストを変えない） |

テストの期待値を変更してよいのは、**ユーザーが明示的に仕様変更を指示した場合のみ**。

### 5. 再テスト

修正後、再度テスト実行。全て通るまで繰り返す。

### 6. E2E テスト（必要に応じて）

E2E テストコマンドが存在する場合は実行する。
