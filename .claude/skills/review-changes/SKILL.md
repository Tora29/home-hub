# コードレビュー

シニアエンジニアの視点で、変更されたコードをレビューする。

## 前提条件

- 実装が完了していること

## ワークフロー

### 1. 変更内容の確認

```bash
git diff
git diff --staged
git log --oneline -10
```

### 2. rules/ の読み込み

`.claude/rules/` 配下のルールファイルを読み込み、プロジェクト固有のレビュー基準を把握する。

### 3. レビュー観点

#### セキュリティ

- [ ] SQL インジェクション
- [ ] XSS（クロスサイトスクリプティング）
- [ ] 認証・認可の漏れ
- [ ] 機密情報のハードコード
- [ ] 入力バリデーション
- [ ] 不審な依存パッケージの追加・バージョン変更
- [ ] ビルド設定ファイルへの依存追加（pom.xml, build.gradle 等）

> 詳細基準: `.claude/rules/security.md`

#### パフォーマンス

- [ ] N+1 クエリ
- [ ] 不要な再レンダリング
- [ ] メモリリーク
- [ ] 大量データの処理

#### ベストプラクティス

- [ ] DRY（重複コード）
- [ ] 単一責任の原則
- [ ] 適切なエラーハンドリング
- [ ] 型安全性

> 詳細基準: `.claude/rules/error-handling.md`

#### プロジェクト規約

- [ ] CLAUDE.md のルール遵守
- [ ] ディレクトリ構造
- [ ] 命名規則
- [ ] コメント・ドキュメント

> 詳細基準: `.claude/rules/` 配下の各ルールファイル

#### UI / コンポーネント設計（FE 変更がある場合）

- [ ] コンポーネント分割基準の遵守
- [ ] アクセシビリティ要件
- [ ] テスト用セレクタの維持

> 詳細基準: `.claude/references/ui-patterns.md`, `.claude/rules/data-testid.md`

### 4. フィードバック形式

```markdown
## レビュー結果

### [CRITICAL] 修正必須

- [ファイル:行] 問題の説明

### [WARNING] 改善推奨

- [ファイル:行] 改善提案

### [OK] 良い点

- 良かった点の説明
```

## 注意事項

- 批判ではなく建設的なフィードバック
- 理由と改善案をセットで提示
- 小さな問題より重大な問題を優先
