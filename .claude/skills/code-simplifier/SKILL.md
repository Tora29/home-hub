# コード簡潔化

コードを分析し、シンプルで保守しやすい形に改善する。

## 前提条件

- 機能が完成していること（テストが通る状態）

## ワークフロー

### 1. 対象の特定

- 指定されたファイル/ディレクトリを確認
- 指定がない場合は最近変更されたファイルを対象

### 2. 分析観点

#### 重複コード

- コピペされたロジック
- 類似パターンの繰り返し
- 共通化可能な処理

#### 不要な抽象化

- 過剰なラッパー関数
- 使われていないインターフェース
- 1箇所からしか呼ばれないヘルパー

#### 使われていないコード

- 未使用の変数・関数
- 到達不能なコード
- コメントアウトされたコード

#### 過剰な複雑さ

- 深いネスト
- 長い条件分岐
- 不要な中間変数

### 3. 改善の実施

| 問題           | 対処               |
| -------------- | ------------------ |
| 重複コード     | 共通関数に抽出     |
| 過剰な抽象化   | インライン展開     |
| 未使用コード   | 削除               |
| 深いネスト     | 早期リターン       |
| 肥大化した画面 | コンポーネント分割 |

### 4. 確認

`specs/infra-spec.md` またはプロジェクトの設定から型チェック・テストコマンドを判断し、実行して動作確認する。

## 原則

- **YAGNI**: 今必要ないものは作らない
- **KISS**: シンプルに保つ
- **3行の重複 > 早すぎる抽象化**
- 動いているコードを壊さない

## コンポーネント分割

ページコンポーネントが肥大化している場合、`.claude/references/ui-patterns.md` の分割基準に従って分割する。
`data-testid` の命名規則は `.claude/rules/data-testid.md` を参照。

### 手順

1. 現在の行数と責務を確認
2. ui-patterns.md の分割基準に基づき分割
3. テスト用セレクタを全て維持（E2E テスト互換性）
4. 分割後にテスト実行で動作確認

## 注意事項

- 変更前後で動作が変わらないことを確認
- 大きな変更は段階的に
- テストがあることを確認してから変更
